Xano CI/CD

GitHub Actions validates every change in an isolated Xano sandbox — a separate workspace that cannot touch production — and only writes to real branches after the sandbox tests pass. Production deploys additionally pause for a required human approval before v1 is updated.

Why The Sandbox

Every Xano branch shares one workspace database, so pushing a schema change to any branch — even a temporary preflight branch — can affect production data. The sandbox is a separate workspace, so pushing there cannot touch v1. That is why all preflight testing runs in the sandbox instead of an in-workspace branch.

Workflow Responsibilities

PR CI

xano-pr-ci.yml

Validates pull requests entirely in the sandbox. Never writes to a real workspace branch.

Dev

xano-deploy-dev.yml

Runs the sandbox gate, then deploys dev-* to a matching persistent Xano branch if green.

Prod

xano-deploy-main.yml

Sandbox preflight, then a gated deploy job: human approval, push to production, verify.

Cleanup

xano-cleanup-dev.yml

Deletes matching Xano branches when dev-* Git branches are deleted or manually selected.

Nightly

xano-regression-tests.yml

Runs scheduled tests against production without pushing code, catching drift outside normal deploys.

Required GitHub Settings

Type Name Purpose
Secret XANO_AUTH Xano API token used by the CLI profile in GitHub Actions.
Variable XANO_INSTANCE_URL The Xano instance origin, such as https://your-instance.xano.io.
Variable XANO_WORKSPACE_ID The workspace ID where the backend is pushed and tested.
Variable XANO_PROD_BRANCH Optional. Defaults to v1.
Variable XANO_CLI_VERSION Optional. Defaults to latest; pin it when you want reproducible CLI installs.
Environment xano-production Active. A required reviewer must approve before any production write, and only main may deploy to it.

The workflows never hard-code the Xano instance URL or workspace ID. GitHub repository variables and secrets provide that configuration at runtime; the sandbox is not seeded with secrets, so tests must mock or skip external API calls.

Branch Rules

  • main deploys to XANO_PROD_BRANCH (default v1) after the sandbox gate and approval.
  • dev-* Git branches deploy to matching persistent Xano branches after the gate.
  • Pull requests are sandbox-only; they never write to the workspace.
  • Cleanup refuses to delete any branch that does not start with dev-.
  • One shared concurrency group serializes every Xano operation — essential because the sandbox is a per-account singleton.

Guardrails In Place

  • Sandbox isolation: preflight runs in a separate workspace, so prod data and schema are never at risk.
  • Approval gate: the xano-production environment requires a human reviewer before deploy.
  • Branch protection on main: pull request required, force-pushes and deletions blocked, linear history enforced.
  • Post-deploy verification: production tests re-run against v1 after every prod push.

Regression Signal

  • Unit tests (test {} blocks inside each .xs) cover reusable functions, query behavior, middleware, and edge cases.
  • Workflow tests cover end-to-end backend paths, including multi-step API behavior.
  • Both suites run in the sandbox for preflight and against v1 for post-deploy verification and the nightly schedule.
  • Every run uploads JSON results as workflow artifacts.

Files Involved

.github/actions/setup-xano-cli/action.yml installs and configures the Xano CLI.
.github/actions/xano-sandbox-gate/action.yml resets, pushes, and tests in the isolated sandbox — reused by the PR, dev, and main workflows.
.github/scripts/run-sandbox-tests.sh runs the sandbox unit and workflow test suites.
.github/scripts/run-xano-tests.sh runs branch tests for production verification and nightly regression.
.github/workflows/xano-pr-ci.yml validates pull requests in the sandbox only.
.github/workflows/xano-deploy-dev.yml gates then deploys development branches.
.github/workflows/xano-deploy-main.yml gates, awaits approval, then deploys and verifies production.
.github/workflows/xano-cleanup-dev.yml removes obsolete dev branches.
.github/workflows/xano-regression-tests.yml runs scheduled production tests.
